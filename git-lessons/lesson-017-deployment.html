    Up to now, the Git concepts we've been talking about have been pretty universal. Chances are that users aren't accessing production environment files from the Gitlab server(s), though. Those are probably hosted on a production server better attuned to end-user access. We'll consider a couple different possibilities:

    • A network file share
    • Sharepoint

  • Network File Share

    • Clone the latest version of the project to your "Development" folder
    • Test the new version for functionality
    • Make a copy of the entire network folder holding the production files as a backup
    • Copy-paste the changed files from your Development folder to the production network folder, over-writing the old versions
    • Set a calendar reminder to yourself to delete the backup folder as necessary

  • Sharepoint

    • Clone the latest version of the project to your "Development" folder
    • Test the new version for functionality

      • We'll still be copying files from your Development folder to a Sharepoint folder, but we'll have to keep two things in mind:

      • Sharepoint has limitations on the filetypes it will serve:
        • Most notable is the fact that it doesn't serve ".html" filetypes. Those must be renamed to ".aspx" (Thank Microsoft for that)
            • Also: any references between .html files will need to use the extension .aspx. Again, thank Microsoft.

      • Sharepoint is "helpful":
        • If you overwrite an existing filename, Sharepoint tries to "help" you by renaming the old file, and then finding any references to it in any other files on Sharepoint and changing those references to the renamed old file...because obviously that's what you meant to do. So you'll still end up with the old content.
        • To avoid Sharepoint's "help," first delete the old files, then copy the newer versions into their place
