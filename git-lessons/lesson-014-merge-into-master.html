DCN: How to Git

Lesson 14 - Merge into master

  • Note: The following instructions are specific to the Green-side gitlab. These instructions also apply to the Red-side gitlab, however the URL is different, so keep that in mind.
  • To recap where we are:
    • In Lesson 6, we cloned the "hello-git" repository
    • In Lesson 7, we made a new branch to do some work in: "addNiceGreeting"
    • In Lesson 8, we made a change to a file in our local copy of the "hello-git" repository
    • In Lesson 10, we "staged" our branch's files for the next step
    • In Lesson 11, we took a snapshot of our branch by using the "git commit" command
    • In Lesson 12, we pushed changes to the origin (Gitlab)
    • In Lesson 13, we checked to make sure the "addNiceGreeting" branch made it to Gitlab
    • In THIS lesson, we will request that the "addNiceGreeting" branch be merged into the "master" branch...

  • Login to your UNET workstation (or Remote UNET, or UVDI, etc.)
  • Start the Chrome browser and go to "https://gitlab.fbi.gov"
  • Login as necessary
  • In the upper left, click on "Projects"
  • In the dropdown menu that appears, click "Your projects"
  • A list of projects you have access to appears
  • Find the project named "Hello Git" and click on it
  • Under the project title, look at the section showing the number of branches belonging to this project:
      • 1 Commit - 2 Branches - 0 Tags - 72 KB Files - 72 KB Storage
  • Click on the heading for "2 Branches"
  • You'll come to a page showing a listing of "Active Branches"
  • In the entry for the "addNiceGreeting" branch, click on the button labeled "Merge request"
  • Next comes a page to enter a "New merge request"
  • The first line says:
      • From addNiceGreeting into master Change branches
  • As it says, you'll be merging the "addNiceGreeting" branch into the "master" branch
  • You could change those by clicking on the "Change branches" link...we won't do that today
  • The "Title" of this Merge Request defaults to the "memo" submitted with the most recent "commit"
  • You can enter a more detailed description of the features/functionality being added from the "addNiceGreeting" branch into the "Description" field (we'll leave it blank this time)
  • Next comes a series of fields
  • The meanings of these fieldnames have changed over time...but here's a guide:
    • Assignee: this stands for the person with the most vested interest in this Merge Request --- probably you
    • Reviewers: these will be people who will be sent a notification with a link to look over the code changes you have made --- not necessarily "approvers" of the merge
    • Milestone: IDK...I've never used it
    • Labels: IDK...I've never used it
    • Approvals: This one is made up of 2 subfields:
      • Approvers: People who are authorized to approve this merge
      • Approvals required: The number of approvers who must approve the merge
    • Merge options: Another one made up of 2 subfields:
      • Delete source branch...: in our case, this means that, once the changes from "addNiceGreeting" are merged into "master", then the "addNiceGreeting" branch will be deleted --- MAKE SURE THIS IS CHECKED
      • Squash commits: in our case, we only made one snapshot ("commit"). In some cases, a coder might make multiple "commits" in a coding session, then do a final "git push" at the end of the day. The "squash" option will allow you to describe all those commits with a single line of text rather than keeping track of all the others that you don't need to keep track of anymore --- N/A IN OUR CASE
  • Finally, at the bottom of the Merge request, there is a blue button labeled Create merge request"
  • Click that button!
  • Next time, we'll approve that merge request!
  • DONE!
